# Техническое задание: Интеграция Selfwork Эквайринг API

**Версия:** 1.0  
**Дата:** 2026-05-06  
**Платформа:** .NET 9 / ASP.NET Core 9

---

## 1. Цель и область применения

Разработать промышленную клиентскую библиотеку и демонстрационный веб-сервис для интеграции с платёжным API сервиса [selfwork.ru](https://selfwork.ru). Библиотека должна быть пригодна к публикации в NuGet и использованию в любом .NET 9 приложении (ASP.NET Core, Worker Service, Console).

---

## 2. Функциональные требования

### 2.1 Операции API

| # | Операция | Метод | Эндпоинт |
|---|---|---|---|
| 1 | Верификация ключа | `GET` | `/acquiring/verify` |
| 2 | Создание счёта на оплату | `POST` | `/acquiring/invoice` |
| 3 | Получение статуса счёта | `GET` | `/acquiring/invoice/{id}` |
| 4 | Отмена счёта | `DELETE` | `/acquiring/invoice/{id}` |
| 5 | Список счётов | `GET` | `/acquiring/invoices` |

### 2.2 Webhook

- Принимать `POST /api/webhook/payment` с телом `PaymentWebhookPayload`
- Верифицировать подпись `X-Webhook-Signature` (HMAC-SHA256)
- Отклонять невалидные подписи с `401 Unauthorized`

### 2.3 Бизнес-логика демо-сервиса

- Эндпоинт `POST /api/payment/create` — создать счёт, вернуть ссылку на оплату
- Эндпоинт `GET /api/payment/{id}/status` — проверить статус
- Эндпоинт `DELETE /api/payment/{id}` — отменить счёт
- Эндпоинт `GET /api/payment/health` — проверить соединение с API

## 3. Нефункциональные требования

### 3.1 Надёжность

- **Retry-политика:** 3 повтора с экспоненциальной задержкой (1s, 2s, 4s) на HTTP 5xx и таймаутах
- **Circuit Breaker:** открывается после 5 ошибок за 30 секунд, восстанавливается через 60 секунд
- **Таймаут:** настраиваемый, по умолчанию 30 секунд

### 3.2 Безопасность

- API-ключ передаётся только через заголовок `X-Api-Key`, никогда в URL
- Webhook-секрет хранится только в конфигурации, не логируется
- Все входящие данные валидируются через Data Annotations

### 3.3 Наблюдаемость

- Structured logging на каждой операции: запрос, ответ, ошибки
- `ILogger<T>` — без Console.WriteLine и Debug.WriteLine
- Уровни: `Information` для успешных операций, `Warning` для ретраев, `Error` для сбоев

### 3.4 Тестируемость

- Все внешние зависимости за интерфейсами (`IAcquiringService`, `IWebhookVerifier`)
- HTTP-слой мокируется через WireMock.Net (без реальных вызовов в тестах)
- Покрытие: happy path + error cases + retry логика + webhook верификация
- Все тесты следуют паттерну **AAA (Arrange / Act / Assert)**:
  - Каждый тест содержит три явных блока с комментариями `// Arrange`, `// Act`, `// Assert`
  - Имена тестов: `Метод_Сценарий_ОжидаемоеПоведение`
  - `RetryCount=0` в тестовой конфигурации для скорости и детерминизма
